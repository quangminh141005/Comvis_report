\section{Methodology}

\subsection{System Overview}
Our receipt OCR system is designed to automatically extract text information from receipt images captured by mobile phone cameras. The system follows a two-stage architecture: image preprocessing and text recognition. Figure \ref{fig:system_architecture} shows the overall flow of the system.

\begin{figure}[htbp]
\centering
\includegraphics[width=0.9\textwidth]{images/System_Architechture.png}
\caption{Overall system architecture of the receipt OCR system}
\label{fig:system_architecture}
\end{figure}

\subsubsection{System Architecture}

The system consists of three main components:

\begin{enumerate}
\item \textbf{Input Module}: Accepts receipt images in common formats (JPG, PNG) from various sources such as smartphone cameras or document scanners.

\item \textbf{Preprocessing Module}: Cleans and enhances the input images through a series of image processing techniques. This module is crucial because raw receipt photos often have issues like blur, poor lighting, and noise that can significantly reduce OCR accuracy.

\item \textbf{OCR Module}: Extracts text from the preprocessed images and outputs the recognized text in a structured format.
\end{enumerate}

\subsubsection{Processing Pipeline}

The complete processing workflow is implemented in the \texttt{ReceiptOCRPipeline} class and follows these steps:

\begin{enumerate}
\item \textbf{Image Loading}: The system loads the input receipt image using OpenCV's \texttt{imread()} function.

\item \textbf{Preprocessing}: The image goes through a configurable sequence of preprocessing steps. The default pipeline includes:
\begin{itemize}
    \item Grayscale conversion (with blue-ink detection)
    \item Denoising (bilateral filter)
    \item Contrast enhancement (CLAHE)
    \item Adaptive thresholding
    \item Morphological cleanup
\end{itemize}
\end{enumerate}

\begin{enumerate}
\item \textbf{Image Loading}: The system loads the input receipt image using OpenCV's \texttt{imread()} function.

\item \textbf{Preprocessing}: The image goes through a configurable sequence of preprocessing steps. The default pipeline includes:
\begin{itemize}
    \item Grayscale conversion (with blue-ink detection)
    \item Denoising (bilateral filter)
    \item Contrast enhancement (CLAHE)
    \item Adaptive thresholding
    \item Morphological cleanup
\end{itemize}

Each step is applied sequentially, with the output of one step feeding into the next. This modular design allows us to easily add, remove, or reorder preprocessing steps based on the receipt type.

\item \textbf{Text Extraction}: The cleaned binary image is passed to the OCR engine for text recognition.

\item \textbf{Output Generation}: The extracted text is returned as a string, which can be further processed or saved to a file.
\end{enumerate}

The modular design of our system makes it easy to maintain and extend. Each preprocessing technique is implemented as a separate function in the \texttt{src/preprocessing/} directory, and the OCR engine is wrapped in a dedicated module. This separation of concerns allows us to:

\begin{itemize}
\item Test individual components independently
\item Swap between different OCR engines if needed
\item Customize the preprocessing pipeline for different receipt types
\item Easily add new preprocessing techniques
\end{itemize}

The entire pipeline can be configured through a configuration dictionary, allowing users to customize the preprocessing steps and OCR parameters without modifying the source code. This flexibility is important because different types of receipts (thermal vs. printed, color vs. black-and-white) may require different preprocessing strategies for optimal results.

\subsection{Image Processing}

Before we can extract text from receipt images using OCR, we need to clean up and enhance the images first. Receipt photos taken with phones often have problems like poor lighting, blur, noise, and low contrast. Our preprocessing pipeline solves these issues step-by-step to prepare clean images for the OCR engine.

\subsubsection{Converting to Grayscale}

The first step is converting color images to grayscale. However, not all receipts are the same - some use black ink while others use blue ink (common in printed invoices). Regular grayscale conversion doesn't work well for blue-ink receipts because it reduces the contrast between text and background.

To handle this, we first check if a receipt has blue ink by comparing the blue and red color channels. If the average blue intensity is more than 10 units higher than the red intensity, we classify it as a blue-ink receipt.

For blue-ink receipts, we use a special conversion formula that ignores the blue channel and only combines the red and green channels:
\begin{equation}
I_{\text{gray}} = 0.5 \times R + 0.5 \times G
\end{equation}

This gives us better text contrast. For normal black-ink receipts, we just use the standard OpenCV grayscale conversion.

\subsubsection{Removing Noise}

Receipt images, especially from phone cameras, often have noise (random specks and grain) that can confuse the OCR. We use two different denoising methods:

\paragraph{Bilateral Filter}
This is our main denoising method. It smooths out noise while keeping text edges sharp - which is exactly what we need. Unlike a simple blur that makes everything fuzzy, the bilateral filter is smart enough to blur noise but preserve important edges like text strokes. We set the filter diameter to 9 pixels and use strength values of 75 for both color and space parameters.

\paragraph{Non-Local Means (NLM) Denoising}
For really noisy or low-quality images, we can use this stronger denoising method. It works by finding similar patterns throughout the image and using them to reduce noise. We adjust the filter strength ($h$) between 5-10 for normal images and 10-20 for very noisy ones.


\subsubsection{Sharpening}

Phone camera photos can be slightly blurry due to motion or focus issues. To fix this, we apply a sharpening filter using a small $3 \times 3$ kernel:

\begin{equation}
K_{\text{sharpen}} = \begin{bmatrix}
0 & -1 & 0 \\
-1 & 5 & -1 \\
0 & -1 & 0
\end{bmatrix}
\end{equation}

This kernel enhances edges and makes text look crisper by making each pixel more distinct from its neighbors.

\subsubsection{Improving Contrast}

Many receipt photos have poor lighting - either too dark, too bright, or uneven lighting across the image. We use a technique called CLAHE (Contrast Limited Adaptive Histogram Equalization) to fix this.

CLAHE works by dividing the image into small tiles (8$\times$8 pixels) and improving the contrast in each tile separately. This is better than adjusting the whole image at once because different parts of the receipt might need different adjustments. We set the clip limit to 2.0 to prevent the algorithm from amplifying noise too much. This technique works especially well for faded thermal receipts.

\subsubsection{Converting to Black and White (Binarization)}

OCR works best with pure black text on a white background, so we need to convert our grayscale image to binary (only black and white pixels, no gray). We use two methods depending on the image quality:

\paragraph{Otsu's Method}
This technique automatically finds the best threshold value to separate text from background. It analyzes the image histogram and picks the threshold that gives the cleanest separation. This works great for receipts with even lighting.

\paragraph{Adaptive Thresholding}
For receipts with shadows or uneven lighting, we use adaptive thresholding. Instead of using one threshold for the whole image, this method calculates a different threshold for each small region (31$\times$31 pixel blocks). Each pixel is compared to the average of its neighborhood minus a constant value ($C=10$). This adapts to local lighting changes and handles tricky lighting situations much better.


\subsubsection{Cleaning Up with Morphology}

The final step uses morphological operations to fix small imperfections in the binary image. Thermal receipts especially can have broken or faded characters that need repair:

\begin{itemize}
\item \textbf{Closing}: This operation fills in small gaps and connects broken parts of letters. It's done by dilating (expanding) then eroding (shrinking) the text. This is useful because thermal printer quality often degrades over time, leaving broken characters.

\item \textbf{Opening}: This removes small noise specks and dots from the background. It works by eroding first (removing small objects) then dilating back (restoring the important parts).

\item \textbf{Dilation}: For very faint text, we can optionally thicken the character strokes by expanding them slightly using a 3$\times$3 kernel. This makes thin or faded text easier for the OCR to recognize.
\end{itemize}

We use small kernels (1$\times$1 or 2$\times$2 pixels) to avoid accidentally merging separate characters together.

By the end of this preprocessing pipeline, we transform messy phone photos into clean, high-contrast black-and-white images that are perfect for OCR processing. Each step addresses a specific problem commonly found in receipt images, and together they significantly improve text recognition accuracy.

\subsection{Text Detection}
\textbf{Purpose:} This component is responsible for locating and generating bounding boxes around text areas, including straight, curved, or rotated text.

\subsubsection{DBNet++ (Differentiable Binarization)}
\textbf{Overview:} A segmentation-based approach highly effective for locating text regardless of orientation or irregular shapes, utilized as the primary detection module in PP-OCRv5.

\textbf{Architecture (PP-OCRv5):}
\begin{itemize}
    \item \textbf{Backbone:} Employs high-performance architectures like \textbf{PP-LCNet} or \textbf{MobileNetV3} specifically optimized for extreme inference speeds on mobile and edge devices, alongside larger networks like \textbf{ResNet-50} for server-side scenarios.
    \item \textbf{Neck (RSE-FPN):} Uses an advanced \textbf{Feature Pyramid Network (FPN)} with \textbf{RSE-FPN} (Residual Squeeze-and-Excitation FPN) modules to fuse multi-scale features and preserve spatial information.
    \item \textbf{Detection Head:} Processes output from the Neck through convolutional and de-convolutional layers to generate:
    \begin{itemize}
        \item \textbf{Probability map:} Indicates text presence at the pixel level.
        \item \textbf{Threshold map:} Dynamically calculates binarization thresholds for varied lighting and contrast conditions.
    \end{itemize}
\end{itemize}

\textbf{Processing and Optimization:}
\begin{itemize}
    \item \textbf{Differentiable Binarization (DB):} Produces a Binary Map to extract sharp, polygon-based text boundaries, optimized using binary cross-entropy (BCE) loss.
    \item \textbf{Geometric Refinement:} To refine text localization and bounding box predictions, core geometric mechanisms are applied:
    \begin{itemize}
        \item \textbf{IoU (Intersection over Union):} Measures the overlap between the predicted box and the ground truth.
        \item \textbf{NMS (Non-Maximum Suppression):} Removes redundant overlapping detections, retaining only the box with the highest confidence score for each text instance.
    \end{itemize}
\end{itemize}

\subsubsection{EAST (Efficient and Accurate Scene Text detector)}
\textbf{Overview:} A U-Net style structure designed for fast and accurate scene text localization, bypassing the need for computationally expensive candidates or word grouping.

\textbf{Architecture:}
\begin{itemize}
    \item \textbf{Feature Extractor:} Utilizes a \textbf{VGG16} backbone (with Batch Normalization) to extract multi-level feature maps from the input image.
    \item \textbf{Feature Merger:} Recovers the spatial resolution of features through non-local merging of top-down and bottom-up layers using bilinear interpolation and concatenation.
    \item \textbf{Output Layer:} Predicts two branches: a \textbf{Score map} (probability of text presence) and a \textbf{Geometry map} (\textbf{RBOX format} including 4 distance offsets and 1 rotation angle).
\end{itemize}

\textbf{Processing and Optimization:}
\begin{itemize}
    \item \textbf{Joint Loss:} Combines \textbf{Dice Loss} (for score map imbalance) with \textbf{Log-IoU} (for bounding box dimensions) and a \textbf{Cosine Angle Loss} (for rotation angle periodicity).
    \item \textbf{Restoration and LANMS:} Decodes the RBOX geometry back into 4-point quadrilaterals followed by \textbf{Locality-Aware NMS (LANMS)} to merge neighboring quadrilaterals from the same text instance.
\end{itemize}

\subsection{Text Recognition}
\textbf{Purpose:} Converts cropped text regions into digital character strings.

\subsubsection{CRNN (Convolutional Recurrent Neural Network)}
\textbf{Overview:} A hybrid architecture that combines CNN-based feature extraction with RNN-based sequence modeling.

\textbf{Architecture:}
\begin{itemize}
    \item \textbf{Convolutional Stage:} A customized \textbf{VGG16-based extractor} with 7 convolutional blocks. Later MaxPool layers utilize a rectangular kernel and stride of \textbf{$(2, 1)$} instead of $(2, 2)$ to preserve horizontal spatial resolution for character sequences.
    \item \textbf{Recurrent Stage:} A deep \textbf{Bidirectional LSTM (Bi-LSTM)} with two layers and 256 hidden units to model contextual dependencies between sequential features.
    \item \textbf{Transcription Layer:} A final linear layer mapping RNN outputs to the target character space.
\end{itemize}

\textbf{Processing and Optimization:}
\begin{itemize}
    \item \textbf{CTC Loss:} Trained using \textbf{Connectionist Temporal Classification} to handle variable-length sequences without requiring precise per-character alignment.
    \item \textbf{Decoding:} During inference, the model utilizes a \textbf{CTC Greedy Search} algorithm to handle blank tokens and consolidate repeated characters for accurate text reconstruction.
\end{itemize}

\subsubsection{SVTR (Single Visual Model for Scene Text Recognition)}
\textbf{Overview:} A specialized transformer-based model for text recognition in PP-OCRv5 that completely replaces Recurrent layers with visual self-attention. To balance high recognition accuracy with low real-world latency, the model adopts an \textbf{SVTR-LCNet} hybrid structure, combining lightweight CNN blocks (PP-LCNet) in the initial stages with SVTR transformer blocks in the final stage.

\textbf{Architecture (PP-OCRv5):}
\begin{enumerate}
    \item \textbf{Progressive Overlapping Patch Embedding:} Converts images into "character components" using two consecutive \textbf{$3 \times 3$ convolutional layers (stride 2)} with Batch Normalization, preserving spatial continuity at character edges.
    \item \textbf{Mixing Blocks:} Interleaves \textbf{Local Mixing} (using a \textbf{$7 \times 11$ sliding window} for stroke morphology) and \textbf{Global Mixing} (capturing inter-character context across the entire image). Local blocks are typically sequenced before Global blocks.
\end{enumerate}

\textbf{Processing and Optimization:}
\begin{itemize}
    \item \textbf{Structural Operators:} Uses \textbf{Merging} operators (\textbf{$3 \times 3$ convolution, stride $(2, 1)$}) to reduce feature map height while preserving width, followed by \textbf{Combining} operators to form a final 1D feature sequence.
    \item \textbf{Prediction:} Features are decoded via a Fully Connected layer combined with \textbf{CTC loss} to resolve the final character string.
\end{itemize}
